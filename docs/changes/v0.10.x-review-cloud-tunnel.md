# v0.10.x Review Cloud Tunnel Plan

## Goal

Complete Review UI cloud tunneling in the 0.10.x line so `opsv review --cloud` can expose the local Review UI through `opsv-cloud` for remote reviewers.

## Current State

### openspec-video

- `opsv review` starts only a local ReviewServer.
- `CloudClient` and `TunnelClient` exist under `src/tunnel/`, but are not wired into `src/commands/review.ts`.
- `CloudClient` expects flat API responses, while `opsv-cloud` currently returns `{ data: ... }` envelopes.
- `TunnelClient` implements the binary WebSocket frame protocol and proxies HTTP requests to the local ReviewServer port.

### opsv-cloud

- Session lifecycle API exists: create, list, status, refresh, close.
- WebSocket tunnel server exists at `/tunnel`.
- Gateway exists at `/s/:sid` and validates review JWTs before proxying requests through the tunnel.
- The tunnel frame protocol is aligned between client and server.
- Server-side production readiness gaps remain: in-memory persistence, limited test coverage, missing global error handling, and security hardening items from the cross-project audit.

## MVP Scope for 0.10.2

The MVP is an internal-beta cloud review flow, not a full production SaaS release.

### User Flow

```bash
opsv review --circle --cloud
```

Expected behavior:

1. Start the existing local ReviewServer.
2. Create a cloud review session with `opsv-cloud`.
3. Connect `TunnelClient` to the returned tunnel session token.
4. Print the local URL, cloud review URL, and session id.
5. Keep the process alive as the current review server does.
6. On TTL shutdown or process signal, close the WebSocket tunnel and close the cloud session.

### MVP Client Work

1. Update `CloudClient` to match the server envelope:
   - unwrap `response.data.data`,
   - map `reviewJwt` to local `jwt`,
   - preserve optional `tunnelUrl`.
2. Add review command options:
   - `--cloud`,
   - `--cloud-url <url>`,
   - `--cloud-api-key <key>`.
3. Resolve cloud configuration from CLI args or environment variables:
   - `OPSV_CLOUD_URL`,
   - `OPSV_CLOUD_API_KEY`.
4. Wire `CloudClient.createSession()` and `TunnelClient.connect()` into the existing ReviewServer startup flow.
5. Add cleanup hooks for:
   - `SIGINT`,
   - `SIGTERM`,
   - server close / TTL shutdown.
6. Improve malformed tunnel request handling so parse failures return an error instead of silently falling back to `GET /`.

### MVP Server Work

For MVP, no protocol rewrite is required. The existing server API and tunnel path can be used as-is.

Minimum server requirements before internal beta:

1. `npm install`, `npm run typecheck`, and `npm run build` pass in `opsv-cloud`.
2. Session creation, gateway, and tunnel can run in one process.
3. API key and session token are not printed in normal production logs.
4. The release note states that 0.10.x cloud review is single-instance internal beta until persistence is added.

## Full 0.10.x Plan

### 0.10.2: MVP Integration

Deliver `opsv review --cloud` with the existing local review flow.

Acceptance criteria:

- Cloud session is created successfully.
- Tunnel connects to `/tunnel?token=...`.
- Public review URL loads the local Review UI.
- Review actions still update local files and manifest state.
- Process cleanup closes the tunnel and session.
- Client build passes.

### 0.10.3: Lifecycle Commands and Diagnostics

Add session lifecycle operations:

```bash
opsv review --cloud --status <sessionId>
opsv review --cloud --refresh <sessionId>
opsv review --cloud --close <sessionId>
```

Add diagnostics:

- better error messages for 401, 402, 429, 5xx,
- tunnel connected/disconnected status,
- session id in logs,
- optional debug logging without leaking API keys or session tokens.

### 0.10.4: Test Coverage and Server Hardening

Client tests:

- `CloudClient` envelope parsing,
- `TunnelClient` PING/PONG and malformed frame behavior,
- review command cloud startup with mocked cloud/tunnel clients.

Server tests:

- session create/status/refresh/close,
- gateway token validation,
- tunnel proxy round trip.

Server hardening:

- global error middleware,
- request body limits on gateway/API paths,
- session creation limiter,
- password hashing for account creation,
- clear production JWT secret requirements.

### Post-0.10.x Production Work

These are not MVP blockers but are required before multi-user production use:

- PostgreSQL persistence for users, sessions, API keys, subscriptions, and access logs.
- Redis-backed tunnel/session state or clear single-instance deployment enforcement.
- structured logging and retention policy.
- OpenAPI or contract tests between `openspec-video` and `opsv-cloud`.
- one-time review link or shorter token handoff to reduce URL-token exposure.

## Non-Goals for MVP

- No provider execution through the tunnel.
- No multi-instance cloud deployment guarantee.
- No full billing/customer portal polish.
- No PostgreSQL/Redis integration in the first MVP pass.

## Verification Checklist

Local client:

```bash
npm run build
npm test -- --runInBand CloudClient TunnelClient
```

Manual E2E:

```bash
# terminal 1, opsv-cloud
npm run dev

# terminal 2, openspec-video project
OPSV_CLOUD_URL=http://localhost:<cloud-port> OPSV_CLOUD_API_KEY=<key> opsv review --circle --cloud
```

Then verify:

1. Open the printed cloud review URL.
2. The Review UI loads through `/s/:sid`.
3. Approve writes `## Approved References` and review history.
4. Design feedback writes `## Design References` and review history.
5. Prompt revision writes review history only.
6. Ctrl+C closes the local server, tunnel, and cloud session.
