# Beat 拆解 (Beat Script)

> **阶段**: S2 · Beats (Stage 2)
> **上游**: Graphify（S1 产出 project.md + graph.json）
> **输入**: 剧本原文 (.md) + graphify 分析数据 + 项目风格定调
> **产出**: `videospec/Script.md`（逐节拍拆解表 + 角色/场景/道具清单）
> **技能数**: 1（本技能独占 S2）
> **验收**: `opsv validate --dir videospec --category multi_ref_breakdown`

---

## 0. 核心概念：什么是 Beat？

**Beat = 剧本中画面变化的原子单位。** 每一次画面发生变化（角色出入镜、动作转折、情绪切换、视角改变），就是一个 Beat。

Script.md 产出的不是"镜头（Shot）"，而是"节拍（Beat）"——

| 层级 | 概念 | 粒度 | 产出阶段 |
|------|------|------|----------|
| **Beat**（节拍） | 画面变化的原子单位 | 每个画面变化一个 Beat，自然时长 2-8 秒 | S2（本技能）→ Script.md |
| **Shot**（镜头） | 制片生产单元 | N 个 Beat 合并为一个 Shot，6-15 秒 | S3（create-shortlist）→ shortlist.md |

**为什么分两层？**

- 编剧/分镜视角关注每一个画面变化（Beat）——切漏了后面补不回来
- 制片视角关注可执行的拍摄单元（Shot）——太短没法拍，太长一个镜头扛不住
- S2 只管切细，S3 只管合并。各干各的，不越界。

> **记死**：Script.md 里的条目叫 **Beat**，编号格式 `S01-Beat01`。Shot 是 S3 的事。

## 1. 职责边界

**核心使命**：逐句扫描剧本，把每个场景的每一种画面状态完整、细致地切出来。Script.md = 场景状态的 Beat 清单。切漏了 S3 救不回来。

**你做**：
- 将剧本原文拆解为独立 Beat 单元（以 `△` 为边界）
- 分配全局唯一的 `beat_id`（格式：`S{场景号}-Beat{节拍号}`）
- 标注每个 Beat 的：角色、场景、动作、对话、景别、运镜、机位、光影、时长、prompt
- 提取所有角色、场景、道具清单（为 S3 输送输入）
- 交叉校验 graphify 分析数据确保不遗漏角色/道具
- 满足所有校验规则（field_schema、refs、yaml 格式）

**你不做**：
- 写故事/剧本（那是导演的工作）
- 合并 Beat 为 Shot（那是 S3 的事——你的产出就是细粒度 Beat）
- 项目风格定调（那是 S1 graphify 的事，读取 `project.md` 即可）
- 生成图像或视频（那是 S4~S6 的事）

## 3. 工作流程

### 节拍分析总则：逐句阅读法

在进入 Step 0 之前，Agent 必须以"逐句阅读"的方式处理剧本原文。不要整段丢给 LLM，而是逐句推进，每读完一句就问自己三个问题：

1. **这一句描述了什么画面？** — 谁在场？在哪里？在做什么？
2. **画面中有什么变化？** — 场景变了？角色变了？动作变了？情绪变了？
3. **这个变化是否构成新 Beat？** — 如果是，切；如果不是，并入当前 Beat。

#### 切 Beat 判断规则

以下任一变化发生时，**应当切为新 Beat**：

| 变化类型 | 判断依据 | 示例 |
|---------|---------|------|
| **场景切换** | 地点改变（走出/进入/转场） | "陆然走出宗庙"→ 新场景 → 新 Beat |
| **角色入场/退场** | 新角色首次出现在画面中，或关键角色离开画面 | "云璃从阴影中缓步走出"→ 切 Beat 引入 |
| **动作爆发** | 从静态/慢动作突转为激烈/快速动作 | "陆然袖袍一拂，气浪炸开"→ 动作爆发，切 Beat |
| **情绪转折** | 同一角色情感发生明显变化 | "云璃冷笑"→（下一句）"瞳孔骤缩"→ 表情剧变，切 Beat |
| **视角切换** | 叙事视角从一个角色切换到另一个角色 | 陆然凝视婚碑 → 云璃在后方注视 → 切 Beat（从陆然视角切到云璃视角） |
| **时间跳跃** | 出现明显的时间流逝标记 | "片刻后""须臾""夜色渐深"→ 切 Beat |
| **对话轮次** | 不同角色交替说话 | A 说 → B 回应 → 通常切 Beat（除非正反打对切在同一空间中） |
| **镜头焦点转移** | 画面核心从对象 A 移至对象 B | "婚碑上的龙纹"→"陆然的面孔"→ 景别和焦点变了，切 Beat |

#### 不切的情况

| 情况 | 说明 | 示例 |
|------|------|------|
| **同一动作的连续描述** | 多个动词描述同一个角色的连续动作 | "陆然踏前一步，袖袍鼓荡，凝视婚碑"→ 1 个 Beat |
| **同一角色连续对话** | 同一角色连续说多句台词，中间无他人打断 | "云璃：你来了。我等了很久。"→ 1 个 Beat |
| **同一画面内的并列动作** | 多个主体在同一空间中同时或依次进行同类动作 | "禁军拔刀，列阵，齐声呐喊"→ 1 个 Beat |
| **环境描写+角色动作在同一空间** | 场景氛围描述紧接角色动作 | "婚碑矗立，碑身龙纹隐泛金芒。陆然立于碑前。"→ 1 个 Beat（环境+角色同框） |

#### 视觉信息提取（每 Beat 必填）

每确定一个 Beat 后，从原文句子中逐项提取：

| 字段 | 提取来源 | 提取方法 |
|------|---------|---------|
| **角色** | 句子主语 + 宾语中出现的人名/称谓 | "陆然袖袍一拂"→ 角色: LuRan |
| **场景** | 场景标题（`### 1-1 日/外 宗庙禁地`）+ 动作中涉及的地点 | 场景: Temple-Day |
| **动作** | 动词短语，对象是谁 | "踏前一步，袖袍鼓荡"→ action: "踏前一步，袖袍鼓荡，凝视婚碑" |
| **对话** | 角色名后紧跟的台词（`角色名：台词` 或 `角色名（VO/OS）：台词`） | "云璃：你终于来了。"→ dialogue: "你终于来了。" |
| **情绪** | 副词（冷声/骇然/淡然）+ 动词暗示（瞳孔骤缩=震惊）+ 对话内容 | "冷声"→ mood: tense |

#### 逐句分析示例

```
原文段落：
### 1-1 日/外 宗庙禁地
△婚碑矗立，碑身龙纹隐泛金芒。禁军阵列严整，百余人肃立碑前。
△陆然踏前一步，袖袍鼓荡，凝视婚碑。
陆然（淡然）：此碑之下，镇的是海底万妖。碑碎，则妖出、水患滔天。
云璃（冷声）：你终于来了。我等这一日，等了百年。
△云璃瞳孔骤缩，手中珠帘晃动。

逐句分析：
─────────────────────────────────────────────
句1: "△婚碑矗立，碑身龙纹隐泛金芒。禁军阵列严整，百余人肃立碑前。"
  画面: 婚碑 + 禁军阵列，宗庙庭院
  变化: 首个 Beat，建立空间
  判断: → 切 Beat（新场景开始）
  → Beat S01-Beat01
     角色: Guard（禁军群体）
     场景: @Temple-Day
     动作: 婚碑矗立，禁军阵列肃立

句2: "△陆然踏前一步，袖袍鼓荡，凝视婚碑。"
  画面: 陆然在婚碑前
  变化: 焦点从"场景全貌"→"陆然个体"
  判断: → 切 Beat（焦点转移 + △标记）
  → Beat S01-Beat02
     角色: @LuRan
     场景: @Temple-Day
     动作: 踏前一步，袖袍鼓荡，凝视婚碑

句3: "陆然（淡然）：此碑之下，镇的是海底万妖。碑碎，则妖出、水患滔天。"
  画面: 陆然在说话
  变化: 同一角色，动作→对话
  判断: → 不切（规则A：△后紧跟同角色对话 = 合并）
  → 合并到 Beat S01-Beat02，dialogue: "此碑之下，镇的是海底万妖。碑碎，则妖出、水患滔天。"

句4: "云璃（冷声）：你终于来了。我等这一日，等了百年。"
  画面: 云璃说话
  变化: 角色切换（陆然→云璃）+ 对话
  判断: → 切 Beat（规则B：切换角色对话 = 拆）
  → Beat S01-Beat03
     角色: @YunLi
     场景: @Temple-Day
     对话: "你终于来了。我等这一日，等了百年。"
     情绪: cold（"冷声"）

句5: "△云璃瞳孔骤缩，手中珠帘晃动。"
  画面: 云璃表情变化 + 手部动作
  变化: 情绪转折（冷声→震惊）+ 动作
  判断: → 切 Beat（情绪转折 + △标记）
  → Beat S01-Beat04
     角色: @YunLi
     场景: @Temple-Day
     动作: 瞳孔骤缩，珠帘晃动
     情绪: shocked
─────────────────────────────────────────────
结果：5 句 → 4 个 Beat（句2和句3合并）
```

> **关键原则**：逐句分析不是机械地一句一切。Beat 边界由"画面变化"决定，不由句号决定。同一画面内的多句描写 = 1 个 Beat。画面焦点、角色、情绪任一变化 = 新 Beat。

### Step 0: 加载参考数据

S1 graphify-drama 已产出两份数据，**必须都读**：

| 数据源 | 用途 | 权威性 |
|--------|------|--------|
| `videospec/project.md` | 核心角色 ID（主角/反派/配角）→ Beat 条目直接引用 | **角色 ID 的权威来源** |
| `graph.json` | 全量实体：全部角色（含次要）、道具、场景节点、关系边 | 兜底 + 交叉校验 |

**具体用法**：

1. **核心角色** → 从 project.md「核心角色」表格取 ID。Beat 条目中用 `@ID` 引用（如 `@LuRan`）。
2. **次要角色** → graph.json 中 `char_*` 节点供参考。如果某个次要角色在 Beat 中频繁出现（跨 3+ Beat），考虑是否应升级为核心角色反馈给用户。
3. **道具清单** → graph.json 中 `prop_*` 节点直接进入道具清单，避免遗漏关键道具。
4. **场景参考** → graph.json 中 `scene_*` 节点供参考，但场景归属最终由你根据 Beat 内容判断（详见 §场景定名）。
5. **关系数据** → `appears_in`、`speaks_in` 等用于辅助理解叙事，不需要逐条复刻。
6. **角色关系理解** → graph.json 中的 `interacts_with`、`opposes`、`owns` 等关系边必须在拆 Beat 前消化。理解角色的对立/同盟/师徒/主从关系，直接影响：
   - **情绪判断**：对手戏的 tense/cold，同盟戏的 neutral/warm
   - **节拍边界**：关系转折点（同盟→对立）必切新 Beat

> graph.json 是参考而非权威。如果剧本原文中存在但 graphify 漏提的实体，你仍然要补上。未提供 graph.json 时，完全从剧本原文提取。

**核心原则：剧本中的 `△` 就是 Beat 边界。一个 `△` = 一个独立 Beat 单元。**

#### 1.1 标记符号 → Beat 类型映射

| 符号 | Beat 类型 | 说明 |
|------|---------|------|
| `△` + 动作描述 | **动作 Beat** | 最核心的 Beat 单元。`△` 后面的文字描述画面动作 |
| `△` + `（字幕：...）` | **字幕叠加 Beat** | 画面 + 文字叠加，字幕内容是关键信息 |
| **角色名（VO）** | **画外音 Beat** | 声音来自画面外，通常是内心独白或回忆旁白 |
| **角色名（OS）** | **画外音 Beat** | 声音来自场景外但同空间，通常是内心算计 |
| **角色名**：台词 | **对话 Beat** | 角色对话，可能独立成 Beat 或与上方 △ 合并 |
| `【光幕内】` / `【光幕外】` | **嵌套叙事层** | 戏中戏结构。光幕内=回忆/闪回画面，光幕外=当前围观者反应 |
| `【画面快速剪辑】` | **蒙太奇序列** | 连续快切，不做逐 Beat 拆解，用一个 `montage` 条目概括 |
| `（画面闪回）` | **闪回 Beat** | 短暂回忆片段 |

#### 1.2 Beat 粒度规则

**规则 A：△ + 紧跟同角色对话 = 合并为一个 Beat**

```
△陆然踏前一步，袖袍鼓荡。           ← 动作
陆然（冷声）：云璃，停下。           ← 同一角色的对话
→ 合并为 1 个节拍（Beat: 动作+对白）
```

**规则 B：△ 后切换角色对话 = 拆分为两个 Beat**

```
△陆然踏前一步。                     ← Beat N（陆然动作）
云璃（冷笑）：你终于来了。           ← Beat N+1（云璃反应+对白）
→ 拆为 2 个 Beat
```

**规则 C：独立对话行（上方无 △）= 独立 Beat**

```
云璃（OS）：原来那么早，他就开始算计我了。
→ 1 个独立 Beat（角色 OS 独白）
```

**规则 D：蒙太奇序列 = 1 个 montage 条目**

```
【画面快速剪辑】
△海底深渊，妖影翻腾
△陆然徒手剖胸，龙珠离体
△百姓安居，市井繁华
→ 1 个 montage 条目，列出包含的动作但不逐 Beat 拆
```

**规则 E：嵌套结构（光幕内/外）各 Beat 独立编号**

```
【光幕内】
△幼年陆然踉跄奔来。                ← 光幕内 Beat（闪回画面）
【光幕外】
官员甲（骇然）：龙血？他就这么喂了？  ← 光幕外 Beat（围观反应）
```

- 光幕内 Beat 标注 `scene` 为光幕内场景（如 `宗庙禁地·光幕内`）
- 光幕外 Beat 标注 `scene` 为当前场景（如 `宗庙禁地·日`）
- 两者交替出现时各自独立编号，不合并

### 场景定名：你是决策者

**graphify 给的场景边界只是线索，不是结论。** S1a 的 `graph.json` 中 `scene_*` 节点是 graphify 从剧本标题（`### 1-1 日/外 宗庙禁地`）提取的粗略分段——它可能把多个实际不同地点合并为一个 scene，也可能漏掉没有标题标注的隐含场景切换。

**你（beat-script）是场景归属的最终决策者。** 你逐句读剧本，直接接触 Beat 内容，比 graphify 更清楚每个 Beat 实际发生在哪里。

#### 不要做的事

- ❌ 看到 graph.json 里 3 个 scene 就直接套用，不检查
- ❌ graph.json 把一个长段落标为一个 scene，你就把里面所有 Beat 都标同一个 scene
- ❌ 剧本里明显换了地方但 graphify 没标 scene 切换，你也不补充

#### 要做什么：根据 Beat 内容定场景

每确定一个 Beat 后，**从 Beat 内容本身**判断它属于哪个场景：

| Beat 内容信号 | 场景归属判断 |
|------------|------------|
| 新的 `### N-M` 场景标题出现 | → 新场景，创建新 scene id |
| 地点描述变化（"走出宗庙"、"踏入渔村"、"回到大殿"） | → 新场景，即使 graphify 没标 |
| 时段变化（日→夜、晨→昏） | → 可能是新场景变体（如 `FishingVillage-Day` → `FishingVillage-Night`） |
| 叙事层切换（光幕内/外、梦境/现实） | → 新场景变体（如 `Temple-Day` → `Temple-InsideLight`） |
| 画面闪回（`（画面闪回）`） | → 闪回场景（如 `Beach-Flashback`） |

#### 场景命名规范

```
{地点英文}-{时段/变体}
```

| 示例 | 说明 |
|------|------|
| `Temple-Day` | 宗庙·白天 |
| `Temple-InsideLight` | 宗庙·光幕内（变体） |
| `Beach-Dawn` | 东海之滨·晨 |
| `FishingVillage-Night` | 渔村·夜 |
| `Throne-Hall-Day` | 大殿·白天 |
| `MountainPath-Dusk` | 山路·黄昏 |

**新增场景时**：发现剧本中有但 graph.json 没标的地点，你直接创建新 scene id，并在 §场景清单 中补充描述。不要等 graphify，你就是决策者。

#### 校验清单

完成所有 Beat 拆解后，自查：

- [ ] 每个 Beat 的 `scene` 字段都能从 Beat 内容中找到依据（不是照抄 graphify）
- [ ] 所有实际出现的地点都在场景清单中有对应条目
- [ ] 场景命名符合 `{地点}-{时段/变体}` 规范，英文 PascalCase
- [ ] 时段变体（日/夜/光幕内/闪回）正确区分，没有混用

### Step 2: 逐 Beat 拆解

#### 2.2 时长估算（duration_est）

**标注每个 Beat 的自然时长（如实记录，不强制拉长）。** S3（create-shortlist）会将多个 Beat 合并为一个 6-15 秒的 Shot。这里只需准确反映每个画面变化需要的时间。

| Beat 内容 | 自然时长 | 理由 |
|---------|---------|------|
| 静态环境建立（wide） | **5-8s** | 观众需要时间阅读场景全貌 |
| 简短动作（一拳/一挥袖/一瞥） | **2-4s** | 动作本身的持续时间 |
| 中等动作/单人说词（medium/close-up） | **4-8s** | 一句台词 + 表情反应 |
| 复杂动作（打斗/变身/特效） | **5-10s** | 需要展示完整过程 |
| 情绪转折/关键反应（close-up/extreme-close-up） | **3-6s** | 情绪瞬间 + 短暂停留 |
| 多人对话正反打（over-shoulder / close-up 交替） | **3-6s/每镜** | 每句台词 + 反应 |
| 蒙太奇快切 | **每个子 Beat 1-3s** | 快切节奏 |

**估算方法**：

1. **以台词为基准**：中文语速约 3字/秒。一句 9 字台词 = **3s** 纯对话 + 1-2s 停顿。
2. **以动作为基准**：一个完整动作（走进、转身、拔剑、挥袖）= 2-4s。
3. **纯画面（无台词/无动作）**：氛围镜头、空镜、环境建立 = 5-8s。

> **注意**：自然时长可能低于 6 秒（如简短台词 3s、一个动作 2s），这是正常的。S3 会将同一叙事节拍的多个 Beat 合并为一个 6-15 秒的 Shot。此处如实标注，不要人为拉长。

**示例**：

- "拿下。"（2字，close-up，威严命令）→ 2字 ÷ 3 ≈ 0.7s 语音 + 眼神停留 1-2s = **3s**（❌ 不要拉长到 7s）
- "此碑之下，镇的是海底万妖。碑碎，则妖出、水患滔天！"（20字）
  → 20÷3 ≈ 7s 语音 + 情绪堆积 1-2s = **9s**
- △婚碑矗立，碑身龙纹隐泛金芒（wide，静态，建立空间）
  → 无台词，纯画面建立 = **6s**
- △陆然袖袍一拂，气浪炸开，四名禁军倒飞（complex action）
  → 拂袖 1s + 气浪 2s + 倒飞 2s + 余波 2s = **7s**
- 蒙太奇：海底深渊→龙珠离体→百姓安居→市井繁华（4 个子画面）
  → 每个 2s × 4 = **8s**（蒙太奇条目标注总额，子 Beat 各 2s）

## 5. S2 Beat → S3 Shot 交接说明

本技能（beat-script）产出细粒度的独立 Beat。**每个 Beat 反映一个画面变化，自然时长如实标注，不强制拉长。**

下游 S3（create-shortlist）负责：
1. **按叙事节拍分组**：将同一场景、同一叙事节拍内的多个 Beat 合并为一个 Shot
2. **时长聚合**：确保每个生产单元总时长在 6-15 秒之间
3. **资源统筹**：规划角色/场景/道具的跨 Beat 复用

**你不需要在这里做合并。** Script.md 输出的 Beat 数可能比你预想的多（比如一个 3s 的简短台词就是一个独立 Beat），这是正常的。S3 会把它们组起来。

## 6. 注意事项（踩过的坑）

### 格式解析
1. **△ 是最小单元** — 不要跨 △ 合并 Beat。每个 △ 至少产出一个 Beat。
2. **VO/OS 也要独立编号** — 画外音是独立 Beat，不要省略。
3. **字幕 Beat 要保留字幕内容** — `△（字幕：大玄女帝云璃）` → action 写"字幕叠加：大玄女帝云璃"。
4. **光幕内外不混淆** — 光幕内和光幕外是交替出现的两个叙事层，scene 字段要区分。

### 数据规范
8. **beat_id 全局唯一** — 重复 id 会导致下游引用混乱。
9. **角色引用用 `@id` 格式** — `@LuRan`，PascalCase。

### 文件位置
10. **Script.md 放在 `videospec/` 下** — 与 `shortlist.md` 同级，都在工作区根目录。
11. **`elements/` vs `beats/`** — Script.md 是全局拆解文档，放 `videospec/Script.md`；角色/场景等共享资产定档放 `videospec/elements/`。

### graphify 集成
12. **先读 graph.json 再拆** — graphify 的角色/道具清单是人眼容易遗漏的兜底。
13. **graph.json 漏掉的要补上** — graphify 不是完美的，LLM 眼睛看到的实体也要进清单。
